# 2026-05-31 首批 Skill 数字员工组合样板

## 摘要

这份样板把首批已验证的公开 Skill，整理成 8 个能直接拿去做培训、咨询、方案库的企业工作流。

筛选原则很简单：
- 只用公开可复核、许可边界相对清楚的条目
- 优先做“有人审、可回滚、可追责”的流程
- 不碰客户数据、价格、Cookie、私有仓库和未公开产品信息
- 能落地到岗位，不写成“万能自治”故事

首批不适合直接进样板的，主要是许可边界不清、风险过高或太像工具仓库本身的条目。比如 `baoyu-skills`、以及一些需要额外法务确认的 copyleft 样本，都不放进这一轮。

## 这批样板怎么用

建议把每个 workflow 当成一张可复用的“岗位动作卡”：
1. 先认定场景和 owner。
2. 再选 Skill 组合，不要先选工具。
3. 先跑一个低风险试点，留人工审核点。
4. 试点跑顺后，再把输入、输出、边界、失败回退写成标准件。

## 1) AI 代码验收工作流

### 适用行业
- 软件/互联网/研发组织
- AI/数据基础设施团队
- 咨询/培训/知识服务（用于交付模板演示）

### 岗位 owner
- 技术负责人
- 研发效能负责人
- QA 负责人
- 代码审阅者

### 输入
- issue 或需求说明
- 分支 diff / PR
- 测试计划
- 风险点说明

### 输出
- 验收结论
- 测试记录
- review 备注
- merge / reject / rework 决策

### 推荐 Skill 组合
- `obra/superpowers`
- `openai/codex`
- `github/github-mcp-server`
- `microsoft/playwright-mcp`

### 人工审核点
- 需求是否写清楚
- 是否改到核心模块或权限相关代码
- 单测、集成测、回归测是否真的跑过
- 是否有自动合并、自动发布冲动

### 风险边界
- 不直接改生产环境
- 不把生成代码当最终答案
- GitHub token scope 要收紧，默认只读优先
- 浏览器自动化只在隔离账号和可回滚环境里跑

### 首个试点建议
- 先拿一个低风险 bugfix 或文档类 PR 做样板
- 让数字员工只做“读仓库 + 跑测试 + 写验收意见”三件事

### 步骤
1. 读取 issue、PR 和相关代码
2. 用 `superpowers` 先把验收标准补齐
3. 用 `codex` 跑局部修复或验证
4. 用 `playwright-mcp` 做必要的网页回归
5. 用 `github-mcp-server` 写 review 结论
6. 交给人工确认后再进入合并流程

## 2) MCP 工具接入工作流

### 适用行业
- 软件/互联网/研发组织
- AI/数据基础设施团队
- 通用企业内部运营

### 岗位 owner
- AI 平台工程师
- 系统集成负责人
- 数字员工架构师
- IT / 安全管理员

### 输入
- 目标系统清单
- 认证方式
- 读写权限需求
- 审计要求
- 数据出入边界

### 输出
- MCP server 接入清单
- 权限矩阵
- 审计要求说明
- 上线 / 不上线建议

### 推荐 Skill 组合
- `modelcontextprotocol/servers`
- `PrefectHQ/fastmcp`
- `googleapis/mcp-toolbox`
- `casdoor/casdoor`
- `github/github-mcp-server`

### 人工审核点
- 哪些 connector 只读，哪些允许写
- Secret 放在哪里，谁能看到
- 是否存在隐蔽导出路径
- 是否有日志和审计开关

### 风险边界
- 默认最小权限
- 不把一个 connector 的权限当成整个流程的权限
- 涉及账号、文档、仓库、工单系统时，要分开审批
- 先做只读接入，再做写入或自动操作

### 首个试点建议
- 先接一个只读 GitHub 或文档检索场景
- 目标不是“会干活”，而是“接得稳、看得见、能回收”

### 步骤
1. 列出目标系统与可触达数据
2. 按系统拆分 connector
3. 定义读写 scope 和审计要求
4. 先做只读 MVP
5. 通过后再加写权限和自动化动作

## 3) 知识库资料清洗工作流

### 适用行业
- 咨询/培训/知识服务
- 内容/媒体/营销服务
- 通用企业内部运营

### 岗位 owner
- 知识库运营
- 文档运营
- 内容研究助理
- 业务支持人员

### 输入
- PDF、Word、PPT、网页链接
- 会议纪要
- 旧 FAQ
- 草稿笔记

### 输出
- 统一格式的 Markdown
- 标签和目录结构
- 引用来源列表
- 红线 / 需复核点清单

### 推荐 Skill 组合
- `microsoft/markitdown`
- `alirezarezvani/claude-skills`
- `langchain-ai/open_deep_research`
- `mem0ai/mem0`

### 人工审核点
- 原始来源是否可靠
- 是否有敏感信息没删干净
- 术语和标签是否统一
- 引用是否真的对应原文

### 风险边界
- 不能把 OCR / 转码结果直接当事实
- 不能把内部资料原样扩散到公开库
- 记忆沉淀前要先做隐私清洗
- 不把社区 skill 整仓照搬进企业知识库

### 首个试点建议
- 先挑一个业务主题，清洗 20 份以内的资料
- 目标是建立一个“统一格式 + 统一标签 + 统一审核”的小样本

### 步骤
1. 收集资料并分类
2. 用 `markitdown` 做格式统一
3. 用 `open_deep_research` 补出处和分歧
4. 用 `claude-skills` 把可复用的模板提炼出来
5. 用 `mem0` 记录可长期复用的术语和结论

## 4) 内容选题到发布工作流

### 适用行业
- 内容/媒体/营销服务
- 咨询/培训/知识服务
- 软件/互联网（面向产品和技术内容）

### 岗位 owner
- 内容运营负责人
- 编辑
- 内容研究员
- 品牌/市场负责人

### 输入
- 选题池
- 目标读者
- 参考资料
- 渠道规则
- 禁用表述清单

### 输出
- 选题排序
- 标题备选
- 文章大纲
- 初稿
- 发布检查清单

### 推荐 Skill 组合
- `langchain-ai/open_deep_research`
- `microsoft/markitdown`
- `browser-use/browser-use`
- `microsoft/playwright-mcp`
- `mem0ai/mem0`

### 人工审核点
- 观点是否站得住
- 事实和数据是否核实过
- 是否触碰平台规则或品牌红线
- 发布前是否做了最后一轮润色

### 风险边界
- 不写客户名、报价、合同、未公开产品信息
- 不把检索结果直接当最终稿
- 不用“很强、很震撼、颠覆式”这种虚词堆内容
- 发布动作必须保留人工确认

### 首个试点建议
- 先做一条固定栏目
- 例如每周 1 篇，连续跑 4 周，先看选题和审核链路能不能稳住

### 步骤
1. 收集选题和资料
2. 用 `open_deep_research` 做初步研究
3. 用 `markitdown` 统一素材格式
4. 用 `browser-use` 辅助抓公开页面信息
5. 用 `playwright-mcp` 做发布前检查
6. 人工确认后发布

## 5) 客服 FAQ / 记忆沉淀工作流

### 适用行业
- 通用企业内部运营
- 零售/电商/客服密集型
- 软件/互联网

### 岗位 owner
- 客户成功
- 客服运营
- 知识运营
- 一线支持主管

### 输入
- 已结单工单
- 高频问答
- 产品更新说明
- 审核过的标准回复

### 输出
- FAQ 条目
- 回复话术库
- 记忆记录
- 升级处理规则

### 推荐 Skill 组合
- `mem0ai/mem0`
- `openai/openai-agents-python`
- `BerriAI/litellm`
- `modelcontextprotocol/servers`

### 人工审核点
- 是否含有 PII 或敏感客户信息
- 是否会把一次性特例写成通用规则
- 是否和最新政策、产品说明一致
- 是否该升级给人工而不是继续自动回复

### 风险边界
- 记忆写入前先脱敏
- 不沉淀未确认的承诺话术
- 对外回复必须保留人工兜底
- 不把“已解决”误当成“可复用”

### 首个试点建议
- 先挑一个产品线或一个支持队列
- 只做高频、低风险、重复性强的问题

### 步骤
1. 收集近期工单和回复
2. 识别高频问题和已确认答案
3. 用 `mem0` 沉淀记忆和短答模板
4. 用 `openai-agents-python` 组织支持型交接流程
5. 用 `litellm` 统一模型出口和日志

## 6) 模型服务成本治理工作流

### 适用行业
- AI/数据基础设施团队
- 软件/互联网
- 通用企业内部平台

### 岗位 owner
- AI 平台工程师
- 成本治理负责人
- 基础设施/运维负责人
- 模型网关负责人

### 输入
- 模型流量
- Token 消耗
- 延迟数据
- 预算阈值
- 失败率和回退日志

### 输出
- 路由策略
- 预算告警
- fallback 顺序
- 月度成本复盘包

### 推荐 Skill 组合
- `BerriAI/litellm`
- `vllm-project/vllm`
- `openlit/openlit`
- `tensorzero/tensorzero`
- `ggml-org/llama.cpp`（离线或边缘补位时）

### 人工审核点
- 路由调整是否会影响关键业务
- 是否有突然切换到低质量模型的风险
- 预算线是否需要重新确认
- 日志里有没有不该保留的输入内容

### 风险边界
- API key 要集中管，但不能让太多人看见
- 不要让成本策略绕过业务验收
- 离线模型和在线模型的能力边界要分清
- 任何自动降级都要有通知和回退

### 首个试点建议
- 先接一个内部助手
- 目标不是极致省钱，而是把“看得见、调得动、能回退”跑通

### 步骤
1. 拉通调用、延迟、成本和失败日志
2. 用 `openlit` 看清链路
3. 用 `litellm` 统一路由和密钥治理
4. 用 `tensorzero` 做实验与策略对比
5. 需要时用 `vllm` 或 `llama.cpp` 做不同部署层的补位

## 7) 网页回归测试工作流

### 适用行业
- 软件/互联网
- 零售/电商
- 内容/媒体（有后台发布系统的场景）

### 岗位 owner
- QA 负责人
- 前端工程师
- 运营自动化负责人
- 测试工程师

### 输入
- 测试脚本
- staging 地址
- 测试账号
- 预期结果

### 输出
- 通过 / 失败记录
- 截图或录屏
- 缺陷单
- 回归备注

### 推荐 Skill 组合
- `microsoft/playwright-mcp`
- `browser-use/browser-use`
- `openai/codex`
- `obra/superpowers`

### 人工审核点
- 是否会触发真实写操作
- 是否在正确的测试环境
- 账号权限是否隔离
- 失败是否来自脚本还是产品本身

### 风险边界
- 不在生产环境乱点
- 不用共享账号跑高敏流程
- 对外发出的真实动作必须有人工审批
- Cookie、会话和截图要按敏感数据管理

### 首个试点建议
- 先做一个最核心的用户路径
- 例如登录、下单、提交、发布四选一，不要一口气铺太多

### 步骤
1. 定义关键路径和预期结果
2. 准备隔离账号和测试数据
3. 用 `playwright-mcp` 跑可视化回归
4. 用 `browser-use` 辅助处理页面状态
5. 用 `codex` 或人工修复脚本
6. 把结果回写到缺陷或验收记录

## 8) 研究简报到可复用草稿工作流

### 适用行业
- 咨询/培训/知识服务
- 内容/媒体/营销服务
- 软件/互联网（产品研究、竞品研究）

### 岗位 owner
- 行业研究员
- 内容研究助理
- 策略分析师
- 知识库运营

### 输入
- 研究主题
- 关键词和限定条件
- 公开来源
- 禁止使用的表述

### 输出
- 来源表
- 结论草图
- 争议点清单
- 可继续扩写的草稿

### 推荐 Skill 组合
- `langchain-ai/open_deep_research`
- `microsoft/markitdown`
- `browser-use/browser-use`
- `alirezarezvani/claude-skills`

### 人工审核点
- 资料是否最新
- 结论和证据是否对得上
- 有没有把推断写成事实
- 是否遗漏反例或边界条件

### 风险边界
- 不把检索结果直接当发布稿
- 不拿单一来源替代交叉验证
- 不写带客户指向的未公开内容
- 引用和转述要留痕

### 首个试点建议
- 先做一份 2 页以内的简报
- 主题最好是已有公开资料、争议不大的方向

### 步骤
1. 定义问题和范围
2. 用 `open_deep_research` 拉资料
3. 用 `browser-use` 补公开网页证据
4. 用 `markitdown` 统一整理
5. 用 `claude-skills` 提炼成可复用结构
6. 人工确认后再扩写

## 组合建议：先跑哪几个

如果想先做一批最容易讲清楚、也最容易培训复制的样板，我建议先跑这 4 个：
1. AI 代码验收工作流
2. MCP 工具接入工作流
3. 知识库资料清洗工作流
4. 内容选题到发布工作流

这 4 个覆盖了研发、平台、知识和内容四条主线，比较适合先做内部培训材料。

如果要再往下补，下一批优先考虑：
- 客服 FAQ / 记忆沉淀
- 模型服务成本治理
- 网页回归测试
- 研究简报到可复用草稿

## 备注

这批样板只写公开能力，不写客户、价格、私有仓库、Cookie、合同或未公开产品信息。后面如果要正式入库，建议把每个 workflow 再拆成单页，补上更细的输入校验、失败回退和审计字段。
