# Taxonomy｜企业场景分类体系

这份分类不是按“工具炫不炫”来分，而是按企业里谁会真的用、会承担什么责任、最容易在哪翻车来分。

## 一级场景

| ID | 场景 | 典型问题 | 当前收录 Skill | 先放哪类数字员工 | 默认边界 |
|---|---|---|---:|---|---|
| office | 办公协同 | 会议纪要、邮件、文档、知识库、汇报 | 3 | 文档助理 / 知识库运营 / 内部助手 PM | 先管原文件权限、记忆保留期、谁能看到什么 |
| finance-tax | 财税 | 发票、报销、预算、成本分析、税务资料准备 | 0 | 财务助理 / 成本分析支持 / 报销流程助手 | 没有复核链路时，不要让代理直接生成对外承诺或税务结论 |
| legal-compliance | 法务合规 | 合同初审、条款对比、隐私合规、风控提示 | 1 | 法务助理 / 合规专员 / 文档处理专员 | 只能做辅助整理和提示，不能替代律师或合规负责人拍板 |
| sales-growth | 销售增长 | 客户研究、线索筛选、拜访准备、跟进、CRM | 0 | 销售研究助理 / 线索运营 / 增长运营 | 别让代理跳过人工判断，尤其不要擅自对客户承诺价格、周期、能力 |
| customer-service | 客服/交付 | 工单归类、FAQ、客户成功、实施手册、复盘 | 1 | 客服系统负责人 / 客户成功运营 / 交付知识库管理员 | 核心风险是记错用户信息、把历史脏数据继续放大 |
| hr-training | 人力/培训 | 招聘 JD、简历筛选、培训课件、员工手册、考核反馈 | 0 | 招聘助理 / 培训运营 / 内部知识运营 | 涉及候选人和员工数据时，先把留存、权限和偏见风险讲清楚 |
| r-and-d | 研发/产品 | 需求、代码、测试、排障、发布检查 | 12 | 开发工程师 / 研发效能 / AI 平台工程师 | 不要神化为“自己写自己发”，spec、测试、review 还得有人扛 |
| marketing-content | 市场/内容 | 选题、写作、图文、视频脚本、分发、数据复盘 | 5 | 内容研究员 / 选题编辑 / 运营自动化负责人 | 内容类最怕像模板稿，先保留人的判断，再谈批量生成 |
| operations-admin | 行政/运营 | 制度、采购、巡检、流程协调、平台接入、账号权限 | 11 | 运营负责人 / 平台运维 / 流程管理员 | 多数工具都能碰账号、日志、文件，默认按高权限风险管理 |
| industry-solutions | 行业方案 | 医疗、教育、制造、政务、园区、金融等垂直场景方案 | 0 | 解决方案经理 / 行业研究员 / 交付架构师 | 没有行业数据和流程细节时，不要空喊“通用解决方案” |

说明：当前样本主要集中在研发/平台/内容三个方向；财税、销售、人力、行业方案暂时还缺可公开复核的代表条目。

## 二级能力

- research / monitoring：先把资料、信号、来源找齐，适合研究助理、情报员、内容研究员。
- drafting / generation：先出初稿或初版，再交给人审，不要直接当最终成品。
- review / compliance / QA：负责找问题、挑风险、做抽检，不负责替人拍板。
- automation / integration：把浏览器、仓库、文件、系统接起来，重点不是“会不会做”，而是“有没有权限做”。
- analytics / memory / reporting：做复盘、记忆、日志、报表，重点是保留期、纠错和可追溯。

## 当前收录的默认判断

1. 能直接碰文件、仓库、浏览器、密钥的 Skill，默认比“只会写字”的 Skill 风险高一档。
2. 看上去最省人的，不一定最适合先落地；企业更适合先上“有人审、可回滚、可追责”的半自动链路。
3. 内容类 Skill 先看有没有人味和事实核查；平台类 Skill 先看权限和审计；记忆类 Skill 先看 retention 和删除机制。
