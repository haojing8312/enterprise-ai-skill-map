# AI 亚马逊评论洞察员 / AI Amazon Review Insights Agent

> 把评论区里的抱怨、夸奖、复购理由，整理成老板能拍板的产品线索。

## 岗位速览

- 适合对象：Amazon 卖家、DTC 品牌、跨境产品团队，尤其是有 5 个以上 ASIN、评论和退货问题开始压到老板桌面的团队。
- 每日输入：ASIN、竞品 ASIN、评论导出、Q&A、VOC/CX Health、退货原因、客服工单摘要、Listing 改动记录。
- 每日输出：老板简报、评论主题表、风险预警、竞品软肋、Listing/FAQ/说明书修复建议、产品改良 backlog。
- 人审边界：不绕过平台限制，不诱导评价，不自动联系客户，不自动改 Listing、价格、广告或合同条款。

## 1. 老板痛点

很多团队的问题不是没数据，是评论躺在后台，开会时没人带证据。运营偶尔截图，客服知道客户在骂什么，产品经理听到的是二手转述。等星级掉下来、退货率上来，老板才发现包装、说明书、尺寸表或卖点表达早就被买家提醒过。

AI 亚马逊评论洞察员要解决的就是这件小而硬的事：每天读评论，把买家原话整理成老板能拍板的行动清单。

## 2. 可接手任务

1. 每日 1–3 星评论分诊：识别安全、合规、批次、尺寸、说明书等紧急问题。
2. 每周 ASIN 评论主题地图：按质量、尺寸、包装、物流、安装、预期落差等标签汇总。
3. 新品上市后 30 天 VOC 追踪：第 7/14/30 天检查首批差评和退货苗头。
4. 竞品差评机会挖掘：低频合规查看或工具导出指定竞品评论，找对手被骂最多的点。
5. Listing 误解修复包：把买家误解转成标题、五点、A+、图片、FAQ 和说明书修改建议。
6. 退货原因 × 评论交叉分析：把后台退货原因和评论主题放在一起看。
7. 客服/售后话术草稿：把高频问题转成待审核回复，不自动发送。
8. 月度产品改良 backlog：把评论痛点转成产品、供应链、包装、说明书任务。

## 3. 能力模块

- 数据接入：Seller Central、Brand Registry、VOC/CX Health、Brand Analytics、合规第三方工具导出、脱敏客服工单。
- 评论清洗：去重、语言识别、字段统一、敏感信息脱敏、短评过滤。
- 主题与情绪打标：按问题类型、情绪强度、影响环节、是否可修复分类。
- 证据引用：每条判断绑定买家原话、链接或 Review ID、日期与样本量。
- 行动排序：按影响收入、修复成本、紧急程度和人工置信度给优先级。
- 老板简报：5 分钟可读，明确本周该改什么、谁负责、哪里需要人审。

## 4. 工作机理

```text
ASIN / 评论 / Q&A / VOC / 退货原因 / Listing 改动记录
        ↓
脱敏、清洗、去重、保留买家原话
        ↓
按质量、尺寸、包装、物流、说明书、预期落差等主题打标
        ↓
提取证据、计算频次、判断风险和机会
        ↓
输出老板简报、Listing 修复建议、产品改良 backlog
        ↓
老板 / 运营 / 产品 / 客服抽样复核后再执行
```

## 5. 可复制提示词

### 提示词 1：低星评论每日分诊

```text
你是 Amazon 卖家的评论洞察分析师。下面是我方 ASIN 的 1–3 星评论数据，字段包括：ASIN、日期、星级、标题、评论正文、变体、Review ID/链接。请只基于我提供的数据分析，不要编造事实。

任务：
1. 按紧急程度把评论分为 P0/P1/P2/P3：P0 是安全、合规、认证、明显产品缺陷、批量同类问题；P1 是可能影响退货率/评分的高频问题；P2 是 Listing 预期不清、说明不清、包装/物流问题；P3 是个别偏好或低证据问题。
2. 每条评论输出：Review ID、ASIN、问题标签、紧急程度、建议动作、是否需要人工复核。
3. 汇总今天最重要的 5 个问题，并给出证据评论原文摘录。
4. 如果样本不足或无法判断，请明确写“证据不足”。
5. 不要建议诱导买家改评价、删评价或提供奖励换评价。

数据如下：
[在这里粘贴脱敏后的评论表格]
```

### 提示词 2：ASIN 评论主题地图

```text
你是跨境电商品类研究员。请分析以下 Amazon 产品评论，输出“方面级评论主题地图”。只使用我提供的数据，不要扩展到未知事实。

请输出 Markdown 表格，列包括：主题、正面评论数量、负面评论数量、典型表达、代表性 Review ID、可能根因、建议动作、置信度（高/中/低）。

主题请优先从这些类别中选择，也可以新增：质量/耐用性、尺寸/尺码、兼容性、安装/使用难度、包装、物流破损、说明书、材质/气味、功能缺失、外观、性价比、客服/售后、Listing 描述不清。

额外要求：
- 区分“产品真实缺陷”和“买家预期/Listing 表达不清”。
- 每个结论必须引用至少 1 条 Review ID 或原文短句。
- 最后给出本 ASIN 最值得优先处理的 5 个问题。

评论数据：
[粘贴评论 CSV 或表格]
```

### 提示词 3：竞品差评机会分析

```text
你是 Amazon 竞品评论洞察顾问。下面有两组数据：A 是我方产品评论摘要，B 是竞品 ASIN 的公开评论或合规工具导出摘要。请找出我方可利用的产品和 Listing 机会。

请输出：
1. 竞品被频繁抱怨的 Top 10 问题；
2. 我方是否已经解决这些问题：已解决/部分解决/未解决/证据不足；
3. 可转化为我方卖点的表达，但不要写夸大、医疗、安全或未经验证的声明；
4. 建议修改的位置：标题、五点、A+、主图/副图、FAQ、说明书、包装、产品结构；
5. 需要产品经理或合规负责人复核的事项。

限制：
- 不要假设我方有未提供的功能。
- 不要建议刷评、诱导评论或批量抓取。
- 所有结论都要引用评论证据。

A 我方数据：
[粘贴我方评论/卖点]

B 竞品数据：
[粘贴竞品评论/摘要]
```

## 6. 人类主管验收

- 老板：每天看 5 条关键发现，确认是否需要拍板。
- 运营：抽查 20 条评论，分类准确率目标不低于 85%。
- 产品经理：检查每个改品建议是否有 5 条以上评论证据。
- 客服主管：确认回复草稿是否合规、准确、没有自动承诺。
- 数据负责人：记录样本量、时间范围、缺失字段和脱敏状态。

## 7. 不能自动化的事

- 不自动联系客户，不请求客户修改、删除或留下正面评价。
- 不自动发补偿、退款、换货、法律、医疗、安全相关回复。
- 不自动修改 Listing、价格、广告预算、库存或品牌内容。
- 不处理未授权买家个人信息；含 PII 的数据要先脱敏。
- 不把 AI 输出当事实结论；必须抽样复核。

## 8. 第一周试点

- Day 0：确认数据来源、权限、脱敏规则和平台边界。
- Day 1：选 1 个主推 ASIN + 2 个竞品 ASIN，整理近 90 天评论和 Q&A。
- Day 2：建立标签体系，AI 初步分类，人工抽查 50–100 条。
- Day 3：输出 Top 5 负面主题、Top 5 正面卖点、代表性原文证据。
- Day 4：生成 Listing、图片、FAQ、说明书和客服话术建议。
- Day 5：开 30 分钟评审会，决定本周 3 个可执行改动。
- Day 6–7：执行小改动，记录下周要验证的指标。
