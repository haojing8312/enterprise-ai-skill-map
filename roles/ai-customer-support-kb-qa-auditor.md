# AI 客服知识库质检员 / AI Customer Support Knowledge Base QA Auditor

> 面向跨境贸易公司老板和客服负责人：这个岗位不负责“自动回复更多客户”，而是每天把 FAQ、产品页、说明书、售后政策和脱敏工单放在一起检查，找出会让公司对外说错话的地方。

## 适合谁

- 已经上线 AI 客服、FAQ bot 或半自动客服的独立站 / Amazon / TikTok Shop / B2B 官网团队。
- 客服、运营、销售经常因为知识库口径不一致返工的公司。
- 有多语言站点、多国家售后规则、代理商/分销商渠道的出海团队。

## 老板痛点

一个客户拿着三张截图来追问：产品页写 7 天可退，FAQ 写定制品不退，英文客服又说运费我们承担。老板怕的不是客服慢，而是同一家公司对外说出三套答案。后面来的可能是差评、赔付、平台投诉，或者代理商拿旧条款谈条件。

## 岗位描述

AI 客服知识库质检员每天接收已批准的知识库、FAQ、产品页、说明书、退款/物流规则和脱敏客服样本，输出知识库质检表、冲突页面、过期答案、缺口问题、AI 客服回归测试题和主管周报。它不直接对外答复客户，不批准退款赔付，不改合同条款，只负责把风险提前摆到人面前。

## 可接手任务

1. 抽样检查 AI 客服回答是否引用正确知识库。
2. 标出 FAQ、产品页、说明书、退款规则之间的冲突。
3. 检查知识库文章是否有更新时间、owner、适用地区、人工升级边界。
4. 从脱敏工单里聚类“客户一直问但知识库没写清”的问题。
5. 业务政策更新后生成 20–30 条回归测试题。
6. 检查多语言知识库的金额、时效、退换货条件是否一致。
7. 整理高风险回答给客服主管：用户问法、AI 回答、证据、建议动作。
8. 每周输出知识库整改看板和复测结果。

## 能力模块

| 模块 | 作用 | 公开工具 / Skill 线索 |
|---|---|---|
| 知识库结构与版本管理 | 管理页面层级、标签、版本、owner、更新时间 | BookStack、Outline |
| 客服样本读取 | 从授权工单/会话中抽样，不暴露客户隐私 | Zendesk Ticketing API、Intercom Conversations API |
| AI 回答评测 | 把脱敏问题变成可重复测试集 | OpenAI Evals、同类 eval workflow |
| 内容质检与风险标注 | 标出过期答案、矛盾页面、敏感承诺、转人工条件 | 质检 rubric、人工审核表 |
| 复测与管理看板 | 记录 owner、截止时间、复测结果 | 表格 / Airtable / Notion / 内部 BI |

## 工作机理

**知识库 / FAQ / 产品页 / 脱敏工单 → 抽样与比对 → 错误分级 → 整改建议 → 人工批准 → 知识库更新 → 回归测试 → 周报复盘**

关键要求：如果证据不足，输出“需人工复核”，不能猜政策；S1/S2 风险必须由人决定；知识库更新后必须用同一批测试题复测。

## 可复制提示词

### AI 回答质检

```text
你是客服知识库质检员。请审查以下用户问题、AI 回答和知识库来源。

输出表格：
1. 判定：通过 / 失败 / 需人工复核
2. 错误类型
3. 严重级别：S1-S4
4. 证据
5. 建议修改
6. 是否需要新增知识库条目

禁止编造不存在的政策；如果证据不足，标记为“需人工复核”。

用户问题：{{user_question}}
AI 回答：{{ai_answer}}
知识库来源：{{kb_excerpt_with_url_and_updated_date}}
业务地区/语言：{{market_language}}
```

### 知识库文章健康检查

```text
请按“客服 AI 可用性”检查这篇知识库文章。

检查项：标题是否清晰、适用范围是否明确、是否有更新时间、步骤是否完整、关键词/同义词是否充足、是否与其他文章冲突、是否写明人工升级边界、是否适合被 AI 引用。

输出：健康分 0-100、主要问题、建议改写后的摘要、建议新增 FAQ 问题、需主管确认的问题。

文章：{{kb_article}}
```

### 工单缺口提取

```text
请从以下脱敏客服工单/会话摘要中提取知识库缺口。

按意图聚类，输出：用户常见问法、现有回答问题、建议新增/更新的知识库标题、优先级、预期影响、需确认部门。

只使用输入内容，不要猜测内部政策。

样本：{{ticket_or_conversation_samples}}
```

### 政策变更回归测试

```text
业务政策发生变更：{{policy_change_summary}}

请生成 10 条 AI 客服回归测试用例。每条包含：用户问法、标准答案要点、必须拒答/升级人工边界、应引用的知识库页面、通过标准。

注意覆盖多种表达方式、异常场景和跨境地区差异。
```

## 人类主管验收

- 今天抽检了多少条回答、多少篇知识库。
- 通过率和事实错误率。
- S1/S2 严重风险是否已暂停引用或转人工。
- Top 知识缺口有没有 owner 和截止时间。
- 多语言页面是否还有金额、时效、退换货条件冲突。
- 更新后的知识库是否通过同一批测试题复测。

## 不能自动化的事

- 退款、赔付、保修例外、账号解封、价格例外。
- 合同条款、平台申诉、法律/税务/监管解释。
- 使用客户个人信息做公开评测或外部模型训练。
- 删除、下线、改写核心政策文章。
- 处理 API token、客户名单、私有报价、供应商条款。

## 第一周试点

- Day 1：选 1 个渠道、1 个主题、1 种语言，例如“网站聊天 + 退换货 + 英文”；定义错误标签和人工升级规则。
- Day 2：盘点 30–50 篇相关文章，检查更新时间、owner、适用地区、重复/冲突页面。
- Day 3：抽取 50–100 条已脱敏 AI 对话或工单，建立通过率、事实错误率、转人工原因基线。
- Day 4：把 Top 问题转成 20–30 条回归测试题，更新或新增 5–10 篇知识库条目。
- Day 5–7：复测同一批测试题，输出整改清单、S1/S2 风险和下周扩展建议。

## 相关 Skill 工具栈

| 工具 / repo / workflow | 用途 | 风险边界 |
|---|---|---|
| [BookStack](https://github.com/BookStackApp/BookStack) | 自托管知识库，管理 FAQ、SOP、版本 | 只放已批准内容；客户数据和私有政策权限隔离 |
| [Outline](https://github.com/outline/outline) | 团队知识库，适合文章 owner、标签、搜索和 API | API token 最小权限；导出前过滤私密 collection |
| [Zendesk Ticketing API](https://developer.zendesk.com/api-reference/ticketing/introduction/) | 授权抽样 ticket、状态、标签、评论 | 只抽必要字段；邮箱、电话、订单号脱敏 |
| [Intercom Conversations API](https://developers.intercom.com/docs/references/rest-api/api.intercom.io/conversations/) | 获取授权会话样本，做缺口聚类 | 仅使用授权 workspace；删除或散列个人标识 |
| [OpenAI Evals](https://github.com/openai/evals) | 把客服问题、标准答案、拒答边界做成回归评测 | 样本脱敏；高风险场景必须人工复核 |

## 风险边界

该岗位只做内部质检、证据整理和回归测试建议。外部回复、退款赔付、保修例外、合同条款、法务合规、平台申诉、敏感客户数据处理必须由人类主管确认。
