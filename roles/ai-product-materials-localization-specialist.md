# AI 产品资料本地化专员 / AI Product Materials Localization Specialist

> 面向跨境贸易公司老板和高管的岗位地图。  
> 本岗位只处理公开产品资料和企业明确授权的已发布资料；不接入客户隐私、合同、报价、成本、未公开新品或供应商内部资料。

## 1. 适合谁

- B2B 制造出口公司：产品目录、官网、说明书、报价前资料长期多版本并存。
- Amazon / eBay / Etsy / Walmart 平台卖家：Listing、A+、图片文字、FAQ 与说明书经常不一致。
- Shopify / 独立站品牌：多语言页面、Google Merchant feed、社媒素材同步成本高。
- 多 SKU、多语言、多渠道运营团队：新品上架前临时翻译，缺术语库、禁用词和发布前检查表。

## 2. 老板痛点

老板最容易忽略的是：海外客户看不懂产品资料，不一定会告诉你。  
他可能直接关页面、买错型号、安装错配件，或者把问题丢给客服。

典型场景：同一个支架在 Amazon 写 12 inch，官网写 30 cm，说明书说适配 A/B/C，FAQ 只写 A/B。广告还在引流，客服每天解释，退货理由却被归成“客户不懂”。这不是单纯翻译问题，而是产品事实在各渠道散了。

## 3. 岗位描述

**AI 产品资料本地化专员**每天接收公开或已授权发布的产品资料：Listing、官网产品页、说明书、FAQ、包装、图片文字、Google Merchant feed、已确认产品事实、目标市场语言、渠道规则和术语库。

它的输出不是“翻译稿”这么简单，而是一组可被运营、产品、客服、合规共同审核的资料包：

- 双语/多语言对照稿；
- 术语库与禁用词表；
- 多渠道冲突清单；
- 发布前 QA checklist；
- 风险标注表；
- 老板版本地化周报。

## 4. 可接手任务

1. 多语言产品资料一致性审计：对比 Listing、官网、说明书、FAQ、包装之间的术语、规格、承诺冲突。
2. 重点 SKU Listing 本地化重写：把机器翻译腔标题、五点、描述改成目标市场自然表达。
3. 产品术语库与禁用词表搭建：统一品牌、型号、材料、功能、认证、保修、单位和渠道字段。
4. 说明书与安装步骤本地化 QA：检查步骤是否容易误解、图片文字是否漏译、警示语是否清楚。
5. FAQ 与故障排查本地化：把购买前疑问和售后问题改成目标语言可读版本。
6. 图片 / A+ / 短视频字幕本地化清单：标记素材编号、原文、建议译文、是否需设计重做。
7. Google Merchant / 独立站 feed 文案 QA：检查标题、描述、属性、单位、货币和落地页是否一致。
8. SKU 上新发布前检查表：把本地化流程沉淀成可复用 checklist。

## 5. 能力模块

| 模块 | 作用 | 可对应的公开工具 / Skill |
|---|---|---|
| 资料资产与版本管理 | 记录 SKU、渠道、语言、URL、资料类型、版本和负责人 | Weblate、表格、NocoDB / Baserow 类看板 |
| 术语库 / 翻译记忆 | 管理品牌术语、品类词、型号、单位、禁用词、标准句式 | Google Cloud Translation Glossaries、DeepL Glossaries、OmegaT |
| AI 初译与本地化改写 | 按目标市场、渠道限制、品牌语气生成草稿 | ChatGPT / Gemini / Claude + 术语库 |
| 质量 QA 与风险标注 | 检查数字、单位、型号、认证、保修、适配、法律风险 | Translate Toolkit、人工 QA 表 |
| 审批与发布交接 | 把草稿交给运营、产品、合规、本地语种人员确认 | CMS / Shopify / Amazon / 内部审批流 |

## 6. 工作机理

```text
输入
  ├─ 公开产品资料：Listing、官网、FAQ、说明书、包装、图片文字、feed 字段
  ├─ 产品事实：规格、尺寸、材料、配件、适配、认证、保修边界
  ├─ 目标市场：国家、语言、渠道、单位、货币、品牌语气
  ├─ 术语资产：术语库、禁用词、历史译文、风格指南
  └─ 安全规则：不接入客户隐私、合同、报价、未公开产品；不自动发布

处理
  ├─ 盘点：按 SKU / 渠道 / 语言建立资料表
  ├─ 抽取：标题、卖点、参数、警示、步骤、FAQ、图片文字、变量
  ├─ 初译：AI / MT 生成目标语言草稿
  ├─ 对齐：术语库、翻译记忆、产品事实、渠道限制
  ├─ QA：数字 / 单位 / 型号 / 认证 / 保修 / 适配 / 法律风险检查
  ├─ 改写：去机器翻译腔，按目标市场习惯重写
  └─ 标注：证据来源、置信度、需人工复核项、版本差异

输出
  ├─ 双语 / 多语言对照稿
  ├─ 术语库与风格指南更新
  ├─ 多渠道一致性冲突清单
  ├─ 发布前 QA checklist
  ├─ 说明书 / FAQ / 图片文字改写建议
  └─ 老板版本地化质量周报

人工验收
  ├─ 运营确认渠道规则和发布位置
  ├─ 产品确认规格、适配、配件、安装步骤
  ├─ 合规/法务确认认证、安全、保修、功效声明
  ├─ 本地语种人员抽样确认自然度和文化表达
  └─ 老板确认优先 SKU、预算和发布节奏
```

## 7. 真实任务示例 / 可复制提示词

### 提示词 1：产品资料本地化 QA 审计

```text
你是跨境品牌的 AI 产品资料本地化专员。请只基于我提供的公开产品资料进行审计，不要补充未提供的事实。

【SKU/型号】{{sku}}
【目标市场/语言】{{market_language}}
【渠道】{{Amazon / Shopify / 官网 / Google Merchant / 说明书 / FAQ}}
【源语言资料】{{source_text}}
【当前目标语言资料，如有】{{localized_text}}
【已确认产品事实】{{facts}}

请输出 Markdown 表格：问题类型、原文/现译文、问题说明、建议改法、风险等级、需谁确认。
检查：误译漏译、术语/型号/尺寸/单位/材质/配件一致性、是否擅自增加功能/认证/保修/功效、目标市场自然度、哪些内容必须人工确认。
```

### 提示词 2：建立 SKU 术语库

```text
你是产品资料本地化术语管理员。下面是某产品线的公开 Listing、官网、说明书和 FAQ。请提取可复用术语库。

【产品线】{{product_line}}
【目标语言】{{target_language}}
【资料文本】{{materials}}

请输出 CSV：source_term,target_term,term_type,preferred_or_forbidden,usage_note,evidence_quote,needs_human_review。
要求：认证、安全、功效、保修类术语标记 needs_human_review=true；不要编造未出现的术语；同一术语多译法要标记冲突。
```

### 提示词 3：Listing 本地化改写

```text
你是目标市场的电商产品文案本地化编辑。请把以下产品 Listing 从 {{source_language}} 本地化为 {{target_language}}，保留事实、自然表达、适合目标市场客户阅读。

【不可改产品事实】{{facts}}
【品牌语气】{{brand_tone}}
【禁用词/禁用承诺】{{forbidden_claims}}
【源 Listing】标题：{{title}}；五点：{{bullets}}；描述：{{description}}

输出：目标语言标题 1 条、五点 5 条、描述 1 段、双语关键术语表、没有翻译或改写的原因、需要人工确认的事实/合规风险。
不得新增未提供的功能、认证、功效、兼容型号、保修承诺；不得使用绝对化词语。
```

### 提示词 4：说明书 / FAQ 本地化检查

```text
你是跨境产品说明书本地化 QA。请检查以下安装说明和 FAQ 是否适合 {{target_market}} 用户理解。

【产品】{{product_name}}
【目标市场】{{target_market}}
【源说明书/FAQ】{{manual_or_faq}}
【产品事实和安全边界】{{facts_and_safety_limits}}

请输出：容易误解的步骤、需要补充图片/编号/警示的位置、单位/尺寸/适配/配件清单是否清楚、更自然的目标语言改写、必须由产品/安全/法务确认的内容。
不要新增安装步骤、配件、适配型号或安全承诺；不确定时写“需要人工确认”。
```

### 提示词 5：老板版本地化周报

```text
你是给跨境老板汇报的 AI 产品资料本地化专员。请把本周的多语言资料审计结果整理成一页周报。

【本周审计范围】{{sku_count, channels, languages}}
【问题清单】{{issues_table}}
【已修订草稿】{{drafts}}
【人工反馈】{{human_feedback}}

周报结构：本周总体判断、Top 5 高风险问题、Top 5 低成本可修复问题、已沉淀资产、下周建议试点范围、数据限制和需老板拍板事项。
要求：简洁中文，适合老板 5 分钟读完；不要承诺销量和排名提升。
```

## 8. 人类主管验收

老板 / 运营 / 产品每天看 6 件事：

1. 本日处理了哪些 SKU、语言和渠道；
2. 发现多少条术语冲突、单位错误、漏译、型号遗漏、风险表述；
3. 每条建议是否有来源链接或原文证据；
4. 高风险项是否被标给产品、合规、法务或本地语种人员；
5. 双语稿是否只基于已确认事实，没有新增功能、认证、保修或功效；
6. 是否出现自动发布、越权上传敏感资料、自动承诺客户结果。

## 9. 不能自动化的事

- 不自动发布 Amazon Listing、官网页面、Google Merchant feed、说明书、包装或广告素材。
- 不自动确认产品事实：尺寸、材料、认证、功效、防水等级、适配、保修条款必须由负责人确认。
- 不处理客户隐私、合同、报价、成本、未公开新品、供应商内部资料。
- 不生成夸大或违规声明：医疗、儿童、安全、环保、认证、功效、绝对化承诺需人工审批。
- 不保证转化率、排名或销量提升。
- 不替代母语审校、法律审查或平台政策判断。

## 10. 第一周试点方案

| 天数 | 工作内容 | 产出 | 验收标准 |
|---|---|---|---|
| Day 0 | 确认 3–5 个 SKU、1 个目标语言、资料来源、禁止范围、审批人 | 试点边界表 | 不接入客户隐私/合同/报价/未公开产品 |
| Day 1 | 盘点 Listing、官网、FAQ、说明书、图片文字 | SKU × 渠道 × 语言资料表 | 每条资料有 URL/文件名/版本/负责人 |
| Day 2 | 抽取术语、规格、型号、单位、认证、禁用词 | 术语库 v1、不可改事实表 | 产品/运营抽查 20 个术语 |
| Day 3 | AI 初译 + 本地化改写 + 多渠道一致性检查 | 双语对照稿、冲突清单 | 每条建议有来源和修改理由 |
| Day 4 | QA：数字单位、适配、警示、功效、保修、自然度 | 风险标注表、需人工复核项 | 高风险内容不进入自动发布 |
| Day 5 | 老板评审：选择可落地修订、沉淀模板 | 一页周报、下周清单、checklist | 指派负责人和发布日期 |

## 11. 相关 Skill / 工具栈

| 工具 / Repo / 公开资料 | 用途 | 风险边界 |
|---|---|---|
| W3C：Localization vs. Internationalization https://www.w3.org/International/questions/qa-i18n | 定义本地化不只是翻译，还包括语言、文化、格式、法律/市场要求 | 不替代各国广告法、标签法规和平台政策 |
| Google Cloud Translation Glossaries https://cloud.google.com/translate/docs/advanced/glossary | 管理术语一致性，约束机器翻译 | 云端翻译前确认资料可外发；高风险内容人工复核 |
| DeepL Glossaries / API Docs https://developers.deepl.com/docs/api-reference/glossaries | 用 glossary 提高术语一致性和自然度 | 不保证法律合规；敏感资料不可默认上传 |
| Translate Toolkit https://github.com/translate/translate | 支持 PO、XLIFF、TMX、TBX 等本地化文件与 QA | 偏技术流程，非技术团队需模板化 |
| OmegaT / Weblate https://github.com/omegat-org/omegat / https://github.com/WeblateOrg/weblate | 翻译记忆、术语、多语言协作、版本管理 | 翻译记忆会继承旧错误；权限和流程需设计 |

## 12. 总编安全结论

可作为群内预览和 GitHub 归档。外部发布仍需郝敬确认。默认试点只使用公开资料或企业明确授权的已发布资料；任何客户、合同、报价、认证、法律、医疗/儿童/食品/安全类信息均需人工复核。
